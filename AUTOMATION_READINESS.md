# AUTOMATION_READINESS

| 루프 | 자동화 수준 | Codex 역할 | Claude Verifier | 위험도 | 권한 | Rollback | 준비 상태 | 보완점 |
|---|---|---|---|---|---|---|---|---|
| 1. 저장소/환경 확인 | HITL | 저장소 상태, 토큰 영향, 기존 파일 유무 확인 | 없음 또는 실행 불가 시 생략 | 높음 | Git 상태 읽기, 토큰 미출력 | 문서/기록만 되돌림 | 조건부 | clone 전 안전 디렉터리, 토큰 비노출, 작업범위 확정 |
| 2. 요구사항/분석 문서 | MANUAL | STEP1_ANALYSIS 정리, 콘텐츠/디자인/게임 요구를 루프화 | 없음 또는 선택적 | 중간 | 읽기 중심, 문서 작성 | 문서 파일만 되돌림 | 준비됨 | 사람 확인이 필요한 콘텐츠를 명시적으로 분리 |
| 3. AORR/TDD 설계 | HITL | 상태머신, retry 규칙, 실패 분류, verifier 정책 설계 | 가능하면 검증, 현재는 실행 불안정 | 중간 | 문서 작성, 제한된 모델 확인 | AORR 문서만 되돌림 | 조건부 | Claude CLI 안정화, 모델명 고정, deploy 승인 절차 확정 |
| 4. 첫 구현 골격 | HITL | `index.html`/`styles.css`/`script.js` 최소 생성 | 실패 시 생략, 실행 불가로 Codex fallback | 중간 | 파일 생성/수정, 로컬 검사 | 3개 핵심 파일과 로그만 되돌림 | 부분 준비 | Claude 검증 복구, 최소 테스트 고정, 루프 경계 명확화 |

지금 자동화하기 가장 좋은 루프 1개: 루프 4. 규칙이 명확하고 변경 범위가 작아서 반복 자동화가 쉽다.

사람이 반드시 승인해야 하는 지점: repo clone/작업 시작, 프로필·게임 규칙 확정, `DEPLOY_APPROVAL_REQUIRED` 단계, Pages 공개 전 최종 승인.

Claude Verifier가 확인해야 할 항목: 파일 존재와 상대 경로, HTML 구조와 링크, CSS 반응형(375/768/1440), JS 오류, 게임 입력/동작, 로컬 HTTP 응답, GitHub Pages 호환성.

Claude 사용 불가 시 Fallback 방식(권한은 모두 허용): Codex가 변경 전 검토와 변경 후 검증을 모두 수행하고, 로컬 파일 검사와 브라우저/정적 응답 확인으로 대체한다.

운영 전 필요한 테스트·권한·모니터링: 로컬 정적 서버 응답, 브라우저 수동 점검, 모바일 폭 확인, `node --check` 계열 문법 확인, Pages 배포 전후 링크 점검, 로그에 fingerprint 남기기.

현업 적용 다음 액션 3개:
1. Claude Verifier를 안정적으로 다시 붙여 baseline/post-change 검증을 표준화한다.
2. Games 영역의 최소 플레이 가능한 snake 골격을 한 루프씩 분리한다.
3. 배포 승인 전용 체크리스트와 롤백 절차를 AORR_LOG 기준으로 고정한다.
